## Overview

Use this when your API, developer tool, or infrastructure product relies on developer adoption but you lack a systematic way to turn docs and tutorials into measurable revenue.

## Integration Tips

**Mintlify / ReadMe / GitBook (Documentation Platform):**
- Use the platform's analytics API to pull page view data into a dashboard (Google Looker Studio or Notion database)
- Configure event tracking: fire a JavaScript event every time a user copies a code block — this is your best activation signal
- Set up "Next Steps" blocks at the bottom of every tutorial linking to the logical next documentation page
- Use Mintlify's or ReadMe's built-in search analytics to identify which queries return zero results — these are content gaps

**Segment / Amplitude / Mixpanel (Product Analytics):**
- Create a "developer journey" funnel: Doc View → Signup → API Key Generated → First API Call → First Successful Response → Integration Complete
- Build developer cohorts by first documentation page visited — compare activation rates by entry point
- Set up automatic Slack alerts when activation rate drops below threshold for any new tutorial-sourced cohort

**HubSpot / Salesforce (CRM):**
- Tag every contact with `first_doc_page` and `last_doc_page_before_signup` properties
- Build a "documentation engagement score" (pages viewed × time on page) and sync to CRM as a lead score modifier
- Create a "developer nurture" workflow that sends tutorial-based emails based on which documentation sections a developer visited (requires bi-directional data sync)

**GitHub (Code & Community):**
- Add Plausible or Fathom analytics to your GitHub README to track clicks-to-docs
- Use GitHub Discussions as a structured Q&A forum that indexes in Google — answer common questions there before they become support tickets
- Create "Code Templates" in GitHub (github.com/[org]/[product]-template-[language]) to make "run this template" the default onboarding path

**Intercom / Drift (Conversational Marketing):**
- Trigger a developer-specific chatbot flow when a user visits the "Error Reference" documentation for the third time — they're stuck and need help
- Configure a "Live Chat" widget specifically for docs that routes to a developer advocate, not a sales rep
- Build an FAQ bot trained on your most common support tickets — this is the highest-ROI automation in developer content

**Zapier / n8n (Automation):**
- Build a workflow: New HubSpot contact with `source=documentation` → Send Slack notification to DevRel team → Trigger personalized tutorial email sequence
- Automate tutorial distribution: New tutorial published → Auto-post to Dev.to → Auto-draft HN Show HN submission → Queue LinkedIn post for dev advocate to approve

## Troubleshooting

**Problem: Tutorials get traffic but almost no signups**
- *Root cause:* Tutorial is solving a "learning" problem, not a "doing" problem. Developers reading out of curiosity won't convert; developers in the middle of building something will.
