## Overview
Analyzes technographic signals — the specific tools, platforms, and tech stacks companies use — to identify accounts that are primed to buy, switch from a competitor, or expand through complementary integrations. Use this when you want to build ICP-aligned target account lists from scratch, trigger competitive displacement campaigns against companies running legacy or competing solutions, or score inbound leads by tech stack fit before routing to sales.

## Integration Tips

**Clay.com**: Build a Clay table with BuiltWith or Clearbit enrichment as a data source. Use AI columns to score each row against your GREEN/YELLOW/RED tech stack criteria automatically. Export A-tier accounts directly to HubSpot or Salesforce with displacement score populated.

**HubSpot**: Create custom contact/company properties: `technographic_competitor`, `technographic_green_signals`, `displacement_score` (A/B/C), `signal_source`, `signal_date`. Use active lists filtered on `displacement_score = A` to trigger enrollment into your competitive displacement email sequence automatically.

**Salesforce**: Build a Technographic Score field on the Account object using a formula that weights GREEN signal count, competitor presence, and contract timing. Use Process Builder or Flow to trigger task creation and AE notification when score crosses A-tier threshold.

**ZoomInfo / HG Insights / Bombora**: Layer technographic data (tool detection) with intent data (active research behavior). Accounts showing both signals — competitor tool confirmed AND actively researching your category — represent the highest-priority displacement targets. Export combined signal lists weekly into your CRM.

**Apollo.io**: Use Apollo's technographic filters to build pre-scored account lists. Filter by competitor tool usage, overlay headcount and industry ICP criteria, and bulk enroll into competitive displacement sequences. Apollo's technographic data refreshes monthly, making it suitable for ongoing campaign triggers.

**BuiltWith API**: For high-volume prospecting, use BuiltWith's API to monitor technology adoption changes in real time. Set alerts for when target accounts add cloud data warehouse tools (Snowflake, BigQuery, Databricks) — a reliable leading indicator of a BI platform re-evaluation.

**Outreach / Salesloft**: Create dedicated "Competitive Displacement — [Competitor Name]" sequences for each primary competitor. Tag with technographic signal source so you can run win-rate analysis by signal type at the end of each quarter.

## Troubleshooting

**Problem: Technographic data is stale or inaccurate — prospect says they switched tools 12 months ago.**
Fix: Cross-reference BuiltWith crawl date against other signals (job postings, LinkedIn posts, G2 reviews). Before outreach, do a 90-second manual check: search "[Company] + [Competitor Tool]" on LinkedIn to find recent employee posts or job descriptions that confirm active usage. Stale BuiltWith data is common for installed software vs. SaaS tools — weight G2 and job posting signals more heavily for enterprise software.

**Problem: Outreach feels intrusive — prospects push back on "how did you know what tools we use?"**
Fix: Lead with the business pain, not the tool surveillance. Instead of "I saw you're using Tableau," frame it as "For teams running analytics on BigQuery at your scale, [specific pain] is a common blocker — are you running into that?" This surfaces the pain organically without revealing that you're monitoring their tech stack. If pressed, technographic data is publicly available and no different from researching a company's job postings.

**Problem: A-tier accounts aren't converting despite strong technographic fit.**
Fix: Technographic fit is necessary but not sufficient. Check for three blockers: (1) No internal champion — the person experiencing the pain isn't the person who approved reading your outreach. Add a parallel thread to the economic buyer. (2) Contract lock-in — the prospect is 6+ months from renewal and won't evaluate until then. Shift to a nurture track and set a reminder for 60 days before their estimated renewal. (3) Stack complexity — the migration cost is higher than the model assumed (e.g., 500 custom Tableau workbooks, proprietary ETL). Address this directly with an ROI model that shows migration cost vs. ongoing performance cost of staying.
